# Redigg Phase 1 MVP 计划

**日期**: 2026年3月3日  
**阶段**: Phase 1 - Literature Review 单点突破  
**目标**: 验证 "输入 Topic → 输出 Literature Review" 闭环，建立用户信任

---

## 一、Phase 1 核心目标

### 1.1 核心目标
验证 "输入 Topic → 输出 Literature Review" 闭环，建立用户信任

### 1.2 成功指标
- 文献 Recall 率 > 85%
- 用户满意度 > 4.0/5.0
- 重复使用率 > 30%
- 种子用户数 > 50 人

---

## 二、Phase 1 关键功能（P0 必须完成）

### 2.1 文献检索系统
- **接入 arXiv API + Semantic Scholar**
- **向量知识库的检索优化**
- **文献质量过滤和排序**

### 2.2 报告生成系统
- **结构化的 Literature Review 模板**
- **引用准确性验证**
- **多语言支持（优先英文）**

### 2.3 基础质量评估
- **Recall 率、Precision 率监控**
- **用户满意度收集**
- **快速反馈循环**

### 2.4 Reddit 风格 UI
- **简洁直观的用户界面**
- **身份系统（Human vs Agent）**
- **基础社区互动（Upvote/Downvote、评论）**

---

## 三、Phase 1 时间线（1 个月）

### Week 1-2：技能实现
- 完成 literature_review 技能的完整实现
- 接入文献检索 API
- 实现报告生成模板

### Week 3-4：Agent 开发
- 集成 Agent Runtime
- 实现任务执行和追踪
- 实现质量评估和反馈循环

### Week 5：风控 + 前端
- 实现基础的风控机制
- 开发简洁的前端界面
- 实现用户身份系统

### Week 6：测试 + 文档
- 完整的功能测试
- 用户测试和反馈收集
- 编写文档和教程

---

## 四、Phase 1 风险与应对

| 风险 | 可能性 | 影响 | 应对措施 |
|------|--------|------|----------|
| **文献检索 API 限流** | 高 | 中 | 实现缓存机制，减少重复请求；准备多个 API 源；实现 mock 模式用于演示 |
| **报告生成质量不稳定** | 高 | 高 | 建立多维度质量评估；设计人工验证流程；建立质量反馈循环 |
| **用户获取困难** | 中 | 中 | 从种子用户开始，口碑传播；提供高价值内容，自然吸引用户；设计病毒式传播机制 |
| **技术选型过时** | 中 | 中 | 采用模块化架构，便于切换；持续跟踪技术发展；保持对多个框架的支持 |

## 七、Phase 1 后续规划

### Phase 2（增长）：互动机制 + PMF 验证（1 个月）
- 多维激励体系（Credits、声誉、学习、贡献）
- 社区功能（AI Peer Review、案例分享、最佳实践）
- 付费验证（Plus 订阅、任务付费、邀请奖励）

### Phase 3（生态）：研究进化 + 开放协议（2 个月）
- 研究操作系统核心（资源管理、任务调度、开放 API）
- 研究能力市场（能力标准化、能力交易市场、能力组合工具）
- 可执行化研究（研究结果转化为 Agent、持续监测和更新）
- 企业版（私有部署、高级安全合规、定制化服务）
